# Software-Sim Gap-Elimination Plan

This document (1) analyzes the gap between the project's current **software-sim**
and a *real* software-in-the-loop device simulation, and (2) lays out a concrete
plan to close the most important missing piece — **device provisioning** — across
the **firmware**, the **web backend** (`hil-simulator`), and the **web frontend**.

---

## 1. What "real software-sim" means

A software-sim (software-in-the-loop) is meant to let you run, exercise, and test
the *whole system* in software with no physical board attached, while behaving
like the real device. A credible software-sim has to cover:

| Capability | Real software-sim expectation |
| --- | --- |
| Behavioral fidelity | Same commands, same wire format, same replies as hardware. |
| Channel / DSP model | Models SNR, noise, demod, BER (for an SDR/RF sim). |
| **Device lifecycle / provisioning** | A new device can be **enrolled**, issued an identity + credential, **claimed/activated**, have its credential **rotated**, and be **revoked/offboarded**. |
| State & observability | Inspectable state, logs, snapshots. |
| Negative paths | Rejects bad input (duplicate enroll, claim before enroll, use after revoke). |
| Parity switch | Same control surface whether hardware or sim is live. |

## 2. Current state vs the gap

What already exists and is solid:

- **Behavioral fidelity** — `firmware-software-sim::gateway::GatewaySim` mirrors the
  on-device command set (toggle downstream, simulated SNMP, deauth, health) and the
  dashboard auto-detects hardware vs simulation.
- **Channel / DSP model** — `hil-simulator/src/sim/{ook,pipeline}.rs` genuinely
  models SNR/noise/threshold and computes BER/CRC.
- **State & observability** — snapshots, command log, live WebSocket monitor.

The decisive gap:

- **There is no provisioning / onboarding lifecycle.** The closest command,
  `CMD_REGISTER_NODE`, only inserts an in-RAM `NodeInfo` with **no identity, no
  credential, no state machine, and no revocation**, and it is explicitly
  *simulation-only* (the hardware backend returns a "not supported" note). ESP-NOW
  peers are hard-coded at flash time (`GATEWAY_MAC`), and TLS certs are static
  files generated by hand. So the software-sim assumes every device is *already
  trusted and keyed* — it skips the single most important first step of a real
  secure system.

**Conclusion:** to make the software-sim "real", it must model the full device
provisioning lifecycle. That is the gap this plan eliminates.

## 3. Design: provisioning lifecycle model

Introduce a device-identity state machine into the existing gateway model so it
flows through all three layers unchanged.

States: `Pending → Active → Revoked` (terminal).

```text
EnrollDevice(device_id, mac)   -> create identity, issue credential, state=Pending
ClaimDevice(device_id)         -> Pending -> Active, onboard as a downstream node
RotateCredential(device_id)    -> reissue credential (Active only)
RevokeDevice(device_id)        -> -> Revoked, offboard the node
```

Each identity carries: `device_id`, `mac`, `state`, a deterministic
`credential_fingerprint` (a fake key/cert fingerprint derived from the device id +
an issuance counter), and `credential_version`.

Negative paths that must be rejected: duplicate enroll, claim of an
unknown/non-pending device, rotate/revoke of an unknown device, and any operation
on a revoked device.

## 4. Plan by layer

### 4.1 Firmware (`firmware/software-sim`)

- Add `ProvisioningState`, `DeviceIdentity` types and a credential-issuing
  registry inside `GatewaySim`.
- Add `GatewayCommand` variants: `EnrollDevice`, `ClaimDevice`, `RotateCredential`,
  `RevokeDevice`.
- Surface `devices: Vec<DeviceIdentity>` on `GatewaySnapshot`.
- Unit tests for the happy path and every negative path.

> Like `RegisterNode`, provisioning is a sim/control-plane concept with no raw
> ESP-NOW equivalent, so the on-device firmware path treats these commands as
> simulation-only — consistent with the existing architecture, and no risky
> changes to the embedded images.

### 4.2 Web backend (`hil-simulator`)

- Re-export the new types/commands from `sim::gateway`.
- `GatewayBackend::Simulation` applies them through `GatewaySim`.
- `GatewayBackend::Hardware`: `command_to_line` returns `None` for provisioning, so
  the existing generic path reports "simulation-only"; add `devices` to the default
  hardware snapshot.
- Tests for enroll→claim→rotate→revoke and rejection cases through the backend.

### 4.3 Web frontend (`web/hil-dashboard`)

- Extend `types.ts`: new `GatewayCommand` variants, `DeviceIdentity`, and
  `GatewaySnapshot.devices`.
- Add a **Provisioning** panel to `GatewayPanel.tsx`: device-id/mac inputs,
  enroll/claim/rotate/revoke buttons, and a device-identity table showing state,
  credential fingerprint, and version.

## 5. Test plan

- `cargo test -p firmware-software-sim` — model + negative paths.
- `cargo test -p hil-simulator` — backend wiring.
- `cargo build -p hil-simulator` — backend compiles with new command surface.
- `npm run build` in `web/hil-dashboard` — frontend type-checks and builds.

## 6. Out of scope (future work, documented not implemented)

- Real cryptographic key generation / CSR / CA signing (currently a deterministic
  fake fingerprint).
- On-device (ESP-NOW) provisioning protocol and NVS-persisted identities.
- Certificate rotation wired into the real TLS 1.3/mTLS ingest path.
- Persisting provisioning state across restarts.
